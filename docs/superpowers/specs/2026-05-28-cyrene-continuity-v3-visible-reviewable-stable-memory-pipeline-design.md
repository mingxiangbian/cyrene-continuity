# Cyrene Continuity v3 Visible Reviewable Stable Memory Pipeline Design

## 目标

把 `cyrene-continuity` 从当前 v2 基础升级为 v3：

```txt
Cyrene Continuity v3:
Visible, Reviewable, Stable Memory Pipeline
```

v3 的目标不是让系统更激进地静默写入长期记忆，而是让系统更主动地记录、总结、提出候选、诊断、整理和提醒，同时把 active long-term memory 的写入权限牢牢放在人工 review gate 后面。

这份 spec 是 v3 umbrella spec。它锁定 v3 的安全边界、架构、现有 bug 修复范围、PR train 和最终验收标准。每个 PR 的具体实现步骤应在后续 implementation plan 中逐步展开，不在本 spec 中提前锁死所有接口细节。

## 非目标

- 不把 Dream 变成 memory authority。
- 不允许 Dream、hook、profile reflection 或 automation 静默把 unapproved memory 写入 active memory。
- 不在 v3 把 SQLite 切换为完整 source of truth。
- 不让 `cyrene_continuity_get` 默认 rebuild index。
- 不实现完整 Phase 4 affect / relationship understanding。
- 不让 strategy overlay 覆盖 Codex personality，也不让它声称 Cyrene 有 subjective emotion。
- 不把 dashboard、project merge、profile patch 一次性实现成重平台。
- 不做 unrelated refactor。只清理会影响 v3 安全边界、reviewability 或 observability 的重复逻辑。

## 当前基础

当前 repo 已经具备 v2 的主要基础：

- Codex plugin manifest。
- MCP server。
- Codex skill。
- optional Stop Hook。
- global/project memory roots。
- pending/promote/reject review hash flow。
- SQLite/FTS memory index。
- Dream `light` / `rem` / `deep-preview` / `deep-apply`。
- profile reflection/apply。
- similar-project hints。
- eval gate。
- doctor command。

v3 不应推倒这些基础。v3 的重点是修复当前 audit 发现的矛盾和风险，然后补齐 pipeline 可见性与 review UX。

## 当前必须修复的问题

### Promote Policy Conflict

当前 policy 文案和实现语义里存在冲突：

```txt
Policy A:
pending -> active requires explicit user approval and review hash.

Policy B:
Dream Deep may auto-promote repeated independent candidates.
```

v3 的规则必须统一为：

```txt
Dream Deep can recommend promotion, but cannot promote unapproved memory.
Active memory requires explicit user approval.
```

这项归入 PR 1。所有 README、skill、doctor、Dream report、CLI output 都必须使用 recommend / recommended-for-review 语义，不再暗示 Dream 可以 auto-promote。

### Misleading Auto-Promote Config

当前 `src/config.ts` 仍有 `memoryAutoPromoteEnabled`，并从 `CYRENE_MEMORY_AUTO_PROMOTE` 读取，默认值是 `true`。这会制造一个虚假的安全开关：用户可能以为 auto-promote 被明确控制，但 v3 不允许 unapproved auto-promote 成为默认能力。

v3 采用 recommend-only 兼容方案：

- `CYRENE_MEMORY_AUTO_PROMOTE` 不再驱动 unapproved active promotion。
- 兼容读取旧变量时必须把它解释为 deprecated，并在 doctor/status 中提示迁移。
- 新语义使用 recommend-only 命名，例如 `CYRENE_MEMORY_RECOMMEND_PROMOTION`。
- 默认值必须安全，且任何配置都不能绕过 approval + reviewHash。

Dream 可以推荐 promotion，但 active promotion 仍必须走 approval + reviewHash。

### Dream Authority Too Strong

Dream 是 maintenance assistant，不是 memory authority。v3 中 Dream 可以：

- 合并重复 pending。
- 规范化 pending metadata。
- 生成 promote/reject/defer recommendation。
- 生成 conflict notes 和 open questions。
- apply user-approved operations。
- apply safe reject/expire operations。
- 维护 tombstone。
- 生成 review artifacts 和 profile patch。
- sync/rebuild SQLite index。

Dream 不可以：

- promote unapproved pending。
- 写 active personal memory。
- 写 active relationship memory。
- 写 active affective memory。
- 把 assistant self-reflection 当作 confirmed user preference。
- 绕过 review hash、validator 或 policy gate。

### Generated Projection Deletion Risk

Projection cleanup 必须只删除带明确 generated header 的文件。即使当前 header 已经是非空字符串，v3 仍必须把以下行为固化为 invariant 和 regression test：

```txt
User-authored MEMORY.md must not be removed by generated projection cleanup.
```

推荐 header：

```txt
<!-- Generated by Cyrene Continuity. Do not edit manually. -->
```

Legacy generated header 也必须是非空显式字符串。任何空 header、missing header 或 ambiguous header 都不能触发删除。

### Node / SQLite Compatibility Risk

当前 `package.json` 允许 Node `>=20`，但 SQLite runtime path 可能依赖较新的 `node:sqlite` 能力。v3 必须让 doctor/status 清楚显示：

- Node version。
- SQLite index availability。
- SQLite unavailable reason。
- fallback mode。
- similar-project retrieval 是否 degraded。

如果正式依赖 `node:sqlite`，应同步收紧 `engines.node`。如果继续支持 Node 20，则 SQLite 必须保持 optional，并提供 JSONL fallback + diagnostics。

### `continuity_get` Read Path Rebuilds Index

当前 `retrieveRoutedMemory()` 会在 `cyrene_continuity_get` 里调用 `adapter.rebuildFromRoots({ roots })`。这把 read path 变成 write path，增加性能、locking 和副作用风险。

v3 规则：

```txt
memory write / approve / reject / dream apply / db rebuild -> sync index
continuity_get -> query index only
```

如果 index stale 或 unavailable，`continuity_get` 返回 diagnostics warning，并使用安全 fallback。它不应默认 rebuild。

### Profile Candidate Semantic Flattening

当前 `activeMemoryFromProfileCandidate()` 会把 profile candidate 的 `type` 固定为 `procedural_rule`，这会丢失语义。v3 必须按 section 映射：

```txt
Project Context         -> project_fact
Interaction Preferences -> interaction_style
Response Policy         -> procedural_rule
Always Apply            -> system_policy
Open Questions          -> open_question
```

同时保留：

- `sourceMemoryIds`
- original evidence summaries
- `reviewHash`
- approval event
- candidate section

### Strategy Output Duplication And Overbroad Challenge

当前 context output 同时包含 `responseStrategy`、`strategy`、`dissent`，语义有重叠。v3 应收敛为更清楚的结构：

```ts
interface CodexStrategyHint {
  tone?: 'direct' | 'technical' | 'careful' | 'firm'
  verbosity?: 'low' | 'medium' | 'high'
  challengePolicy?: 'none' | 'mild' | 'firm'
  avoid?: string[]
  rationale?: string
}
```

Challenge trigger 必须基于明确风险：

- explicit bypass risk。
- explicit safety/privacy risk。
- conflict with confirmed memory。
- conflict with confirmed boundary。
- memory pollution risk。

不能只因为 weak token overlap 就触发 strong challenge。

### Similar-Project Diagnostics Insufficient

Similar-project retrieval 必须始终标记为 hint，而不是 current-project fact。v3 应显示：

- source project id/name。
- similarity score。
- similarity reason。
- included memory reason。
- portability。
- `notCurrentProjectFact: true`。

personal / relationship / affective memories 不得作为 similar-project hint 跨项目转移。把 hint 转为 current project memory 必须需要 explicit confirmation 或 current-project evidence。

### Observability Weakness

v3 要回答一个具体问题：

```txt
Where is the memory pipeline stuck?
```

因此 status、doctor、review CLI、dashboard 不是附加功能，而是 v3 的核心可见性层。

## 核心不变量

- `pending -> active` 必须经过 explicit user approval、`reviewHash` match、validator pass、scope/domain/sensitivity policy pass。
- Dream 可以 recommend promotion，但不能 auto-promote unapproved memory。
- JSONL 是 append-only audit/recovery source of truth。
- SQLite 是 authoritative query/index layer，不是 v3 的完整 source of truth。
- `cyrene_continuity_get` 是 read path，只 query index 或 fallback retrieval，不默认 rebuild index。
- pending candidates 可以作为 provisional hypotheses 出现，但不能作为 confirmed fact 注入。
- similar-project hints 必须标记为 hints，不得成为 current-project facts。
- profile candidates 和 `MODEL_PROFILE.pending.md` 是 review materials，不是 active profile。
- personal / relationship / affective memory 不跨项目转移。
- strategy overlay 只输出 compact hint，不覆盖 Codex personality。
- generated projection cleanup 只能删除带明确 generated header 的文件。

## 目标 Pipeline

v3 的 memory pipeline：

```txt
Codex work
  ↓
Stop Hook writes review-safe session summary
  ↓
Candidate extractor creates pending memory
  ↓
Status/Doctor reports pipeline health
  ↓
Review CLI or MCP shows pending candidates
  ↓
User approve/reject/edit/defer
  ↓
Approved candidates become active memory
  ↓
Index syncs memory.db
  ↓
continuity_get retrieves active memory
  ↓
Codex receives compact continuity context
```

这条 pipeline 的关键不是自动 promotion，而是让每一步都可见、可诊断、可 review、可回滚或可阻断。

## 架构边界

### Capture

`codex hook stop` 只负责生成 review-safe session summary，并在安全条件下提出 pending candidate。它不能 promote、reject、写 active profile，不能把 assistant affective interpretation 写入 active memory。

### Pending Review Queue

`pending.jsonl` 是候选队列。Pending candidate 应逐步补齐：

- `candidate_kind`
- `normalizedKey`
- evidence summary
- risk/sensitivity scores
- recommendation
- review hash
- source/session metadata

`open_question` 永不 active。它只能作为 review/reminder material。

### Review / Approval

MCP approve/reject 和 CLI approve/reject/edit/defer 必须走同一套 review hash、validator 和 policy gate。若 `normalizedKey` 与 active memory 冲突，必须进入显式分支：

- supersede old
- keep both
- reject new

任何 hash mismatch、validator fail 或 policy fail 都返回 conflict/blocked，不 mutate。

### Dream Maintenance

Dream 分层职责：

```txt
Dream Light:
  merge duplicate pending
  normalize formatting
  mark obvious duplicates

Dream REM:
  evaluate pending
  recommend promote/reject/defer
  detect contradictions
  generate open questions

Dream Deep:
  apply user-approved promote operations
  apply safe reject/expire operations
  maintain tombstones
  generate MODEL_PROFILE.pending.md
  sync/rebuild SQLite index
```

如果现有 `deep-apply` 语义仍可能 apply unapproved promotion，则 PR 1 必须收紧。`deep-apply` 的 v3 语义是 apply approved operations，不是 autonomously approve operations。

### Profile Patch Flow

Profile reflection 不直接写 stable `MODEL_PROFILE.md`。v3 使用：

```txt
profile_candidates.jsonl
MODEL_PROFILE.pending.md
MODEL_PROFILE.md
```

Flow：

```txt
session summaries / active memory / review history
  ↓
profile reflection generates candidates and pending patch
  ↓
user review
  ↓
approved structured memory update
  ↓
render stable MODEL_PROFILE.md from structured memory
```

`MODEL_PROFILE.pending.md` 是 review artifact，不是 source of truth。

### Index / Retrieval

Storage policy：

```txt
JSONL:
  append-only audit/recovery source of truth

SQLite memory.db:
  authoritative query/index layer

Markdown projections:
  human review/debug artifacts
```

Write/approve/reject/dream apply/db rebuild 会 sync index。`continuity_get` 只 query index。如果 index unavailable 或 stale，则返回 diagnostics，并使用 JSONL fallback。

### Project Identity / Similar Hints

Project tools 负责防止 project memory 因 projectId 漂移而“丢失”：

```bash
cyrene-continuity codex project status
cyrene-continuity codex project list
cyrene-continuity codex project alias <projectId> <alias>
cyrene-continuity codex project merge <from> <to>
```

Similar-project retrieval 只返回 clearly marked hints，不能改变 current project memory。任何 transfer 都必须走 review。

### Observability

v3 observability 由四类命令构成：

```bash
cyrene-continuity codex memory status
cyrene-continuity codex memory review
cyrene-continuity codex memory dashboard
cyrene-continuity codex doctor
```

这些命令必须回答：

- Stop Hook 是否安装和最近运行。
- session summary 是否写入。
- pending candidate 是否待审。
- recommended promote/reject/defer 是否存在。
- SQLite 是否可用。
- index 是否 stale。
- projectId 是否分裂。
- profile pending patch 是否未处理。
- similar-project hints 是否 degraded 或被 gate 阻断。

## Candidate Classification

新增或规范化 `candidate_kind`：

```txt
project_fact
project_decision
user_instruction
workflow_rule
known_pitfall
rejected_approach
open_question
```

默认 policy：

| Candidate kind | Default scope | Active promotion |
| --- | --- | --- |
| project_fact | project | requires approval |
| project_decision | project | requires approval |
| user_instruction | global/project | requires approval |
| workflow_rule | project | requires approval |
| known_pitfall | project | requires approval |
| rejected_approach | project | requires approval |
| open_question | project | never active |

`candidate_kind` 不取代现有 `domain` / `type`，它补充 review UX 和 policy gate。

## Error Handling

- SQLite unavailable：`continuity_get` fallback 到 JSONL retrieval，并返回 diagnostics；doctor/status 显示原因和 degraded mode。
- stale index：不在 read path 自动 rebuild；返回 warning，并提示 explicit `codex memory db rebuild`。
- review hash mismatch：approve/reject/apply 返回 conflict，不 mutate。
- validator/policy gate fail：返回 blocked reason，不 mutate active memory/profile。
- Dream apply gate fail：只写非 mutation diagnostics/report，不写 active/pending/tombstone/profile。
- unsafe projection target：拒绝写入或删除，不尝试修复用户文件。
- projectId ambiguity：status/dashboard warning；merge/alias 需要显式命令。
- hook failure：记录可见错误，不能让 failed hook 伪装成 successful extraction。

## Final Acceptance Criteria

- User can run one command and know memory pipeline status。
- Stop Hook reliably writes session summaries。
- Session summaries can produce pending candidates。
- Pending memory candidates are visible。
- User can approve/reject/edit/defer pending candidates。
- Dream cannot auto-promote unapproved memory。
- Active memory only comes from approved pending candidates or explicitly approved migration/profile apply flows。
- Active memory is retrieved by `cyrene_continuity_get`。
- Wrong projectId does not leak project-local memory。
- Similar-project memory is clearly marked as hint。
- Profile reflection writes pending profile patch, not stable profile。
- Eval gate blocks unsafe automated changes。
- Projection deletion cannot delete user-authored files。
- SQLite unavailable/stale state is visible to user。
- Config flags are documented, used, and safe by default。
- Codex strategy overlay does not override Codex personality。

## Implementation Planning Notes

后续 implementation plan 应从 PR 1 开始，不应一次性展开 PR 1-10 的全部实现。PR 1 是 v3 的安全前置条件；如果 PR 1 没有完成，后续 dashboard/review/profile work 会建立在不稳定的 promotion policy 上。

每个 PR 的 plan 应包含：

- 需要先写的 regression tests。
- 需要修改的最小文件集合。
- expected RED/GREEN verification。
- commit boundary。
- rollback/blocked condition。

这份 umbrella spec 完成后，下一步应使用 `superpowers:writing-plans` 为 PR 1 写 implementation plan。
